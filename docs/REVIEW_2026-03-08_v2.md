## THE SKEPTICAL PRINCIPAL ENGINEER RE-PASS

### What Actually Improved (Credit Where Due)

1. **The shared IAM role is gone.** This was the single biggest security finding and you killed it properly — 13 dedicated roles, each scoped to exact permissions. The most restrictive roles (adaptive-mode, daily-metrics-compute) have DDB + KMS only. This is real least-privilege.

2. **The DLQ is no longer a black hole.** It classified and retried 4 real messages on first run, which immediately found the todoist stale SECRET_NAME issue. This is the system paying for itself.

3. **The canary is real monitoring.** DDB + S3 + MCP round-trip every 4 hours with per-check metrics. This is what "how do you know it's healthy?" looks like.

4. **The item size guard is pragmatic engineering.** 300KB warn / 380KB truncate with CloudWatch metrics. Lossy but keeps the pipeline running. Correct tradeoff for a personal system.

5. **schema_version + reconciliation job together create a data quality foundation.** You can now version migrations AND detect gaps automatically. These two together are worth more than either alone.

6. **Lambda Layer eliminates the copy-paste deployment problem.** 8 shared modules, 16 Lambdas attached. Fix-once-deploy-everywhere is real now. This was the #1 maintainability finding.

7. **The requirements.txt audit revealed a clean dependency profile.** Only garmin has third-party deps. This is surprisingly good news — supply chain risk is much lower than assumed.

### 15 Harshest But Fairest Critiques (Round 2)

**1. Three critical safety modules are built but not wired. This is worse than not having them.**

`platform_logger.py`, `ingestion_validator.py`, and `ai_output_validator.py` exist as files in lambdas/ but zero Lambdas actually import them. The handover even calls this out under "Wiring Pending (P3)." This creates a false sense of security — the Tier 8 checklist shows them as done, but they provide zero runtime protection today. Someone reading the changelog would believe ingestion is validated and AI output is checked. It isn't.

**2. The deploy/ directory is still 160+ scripts and MAINT-3 is "partial."**

The backup files were cleaned from lambdas/ (good), but deploy/ still contains every historical script ever written. `hotfix_daily_brief_v5.sh` through `v7.sh` are still there. Version-specific deploys (`deploy_v2.72.0.sh`, `deploy_v2.73.0.sh`, `deploy_v2.78.0.sh`, `deploy_v2.79.0.sh`, `deploy_v2.95.0.sh`, `deploy_v2.97.0.sh`, `deploy_v2.98.0.sh`) are still there. One-time fix scripts (`fix_prologue.sh`, `fix_sprite_url.sh`) still there. Calling MAINT-3 "partial" is generous.

**3. Six stale .zip files remain in lambdas/.**

`garmin_lambda.zip`, `habitify_lambda.zip`, `health_auto_export_lambda.zip`, `key_rotator.zip`, `nutrition_review_lambda.zip`, `wednesday_chronicle.zip`. These are deployment artifacts that should live in `deploy/zips/` (which exists!) or be generated on-the-fly. Having them next to source files is how "deploy the wrong zip" incidents happen.

**4. The ARCHITECTURE.md still says `lambda-weekly-digest-role` is shared by 10 Lambdas.**

The IAM section, the Operational Lambdas table, and the IAM Security Model section all still reference the shared role pattern that SEC-1 eliminated. This is exactly the documentation drift the first review warned about. Someone reading ARCHITECTURE.md today gets a wrong security picture.

**5. INFRASTRUCTURE.md is at v2.93.0 — the system is at v3.1.3.**

INFRASTRUCTURE.md still lists 6 secrets (should be 8+), 35 Lambdas (should be 39), and the old Secrets Manager table that doesn't include todoist/notion/dropbox/ai-keys. The Lambda count in the "Lambdas (35)" section is wrong. The EventBridge Scheduler section doesn't reflect the new schedules for data-reconciliation, pip-audit, canary, and DLQ consumer.

**6. The INCIDENT_LOG hasn't been updated since v2.76.1.**

The todoist stale SECRET_NAME incident (caught by the DLQ consumer in v2.99.0) isn't in the incident log. The data-reconciliation first run showing RED (17 gaps) isn't recorded. The incident log is the system's memory — if it stops being maintained, the pattern analysis section becomes stale.

**7. PROJECT_PLAN.md still lists v3.0.0 at the top but the system is at v3.1.3.**

The "Current State" header says v3.0.0, 37 Lambdas. The actual state is v3.1.3, 39 Lambdas, 8 secrets. The completed Tier 8 items still show 🔴 for everything that's actually done.

**8. No CI/CD means every hardening fix is one bad deploy away from being undone.**

You created 13 IAM roles, wired them to Lambdas, split secrets, and updated env vars — all manually. The next time you deploy `daily-brief` with `deploy_lambda.sh`, will it preserve the new role assignment? Or will it revert to the old one? Without CI/CD asserting the desired state, manual deploys can silently undo security hardening.

**9. The Lambda Layer has 16 of 39 Lambdas attached — what about the other 23?**

The Layer covers the email/digest/compute Lambdas, but ingestion Lambdas that import `item_size_guard` or shared utilities still bundle via copy-paste. The strava and macrofactor deploys explicitly fall back if `item_size_guard` isn't bundled (`ImportError → direct put_item`). This fallback pattern means a deploy that forgets to bundle the guard silently disables the protection.

**10. The three "built not wired" modules have no integration test.**

When you wire `ingestion_validator.py` into whoop_lambda.py, how do you verify it doesn't reject valid data? The module has validation rules for 19 sources — if any rule is too strict, it will silently drop good data. There's no test suite for the validator. Same for `ai_output_validator.py` — the BLOCK rules could kill good coaching output if thresholds are wrong.

**11. Secrets are more secure but the secret count is climbing.**

Review #1: 6 secrets ($2.40/mo). Now: 8 active + api-keys in 30-day deletion = effectively 9 ($3.60/mo). The todoist/notion/dropbox split was correct for security, but each new integration will add another $0.40/mo. At 20 sources, secrets alone cost $8/mo — a meaningful fraction of the $25 budget.

**12. The ops dashboard exists but nobody is watching it daily.**

A CloudWatch dashboard is only useful if someone looks at it. For a single-operator system, the dashboard competes with email alerts for attention. The anomaly detector, freshness checker, and now DLQ consumer all send emails — which is the actual alerting mechanism. The dashboard is for investigation, not detection. That's fine, but worth being honest about.

**13. The "wiring pending" items should have been wired before declaring the version done.**

v3.0.0 CHANGELOG says "OBS-1: Structured JSON Logging" and "DATA-2: Ingestion Validation Layer" and "AI-3: AI Output Validation" as shipped features. They're files in a directory, not integrated systems. Shipping the module without the integration is like shipping a fire extinguisher in a sealed box.

**14. The system is now 39 Lambdas for 19 data sources for 1 user.**

That's 2 Lambdas per data source, or ~$0.26/month per Lambda. The cost is fine, but the operational surface area is real. Each Lambda needs: an IAM role, a CloudWatch alarm, a DLQ config, an EventBridge schedule, a deploy script, a requirements.txt, and to be included in future audits. At what point does the Lambda count itself become the maintenance burden?

**15. The hardening was done at sprint speed — verify it stuck.**

10 version bumps in one session is impressive velocity but high risk. The IAM role decomposition, secret split, input validation, Lambda Layer, schema versioning, and 4 new Lambdas were all deployed in rapid succession. The first thing to do next session is verify: invoke each Lambda, check it uses the new role, reads the right secret, and the Layer is functional. Trust but verify.

## DESIGN DECISIONS TO PRESERVE (Updated)

Everything from Review #1, plus:

11. **IAM role per Lambda with graduated restrictiveness.** The most restrictive roles (adaptive-mode, daily-metrics-compute) have only DDB + KMS. This is the right pattern.

12. **item_size_guard with truncation.** Lossy but pipeline-preserving. The 300KB/380KB thresholds with CloudWatch metrics is exactly the right tradeoff.

13. **DLQ consumer with classify-retry-archive pattern.** Found a real bug (todoist SECRET_NAME) on first run. This is the system paying for itself.

14. **Canary with dedicated partition.** `CANARY#matthew` pk isolation prevents test data from contaminating real data. Clean design.

15. **pip-audit with first-Monday guard.** Using EventBridge for weekly schedule + in-Lambda day-of-month guard for monthly execution is clever and avoids a separate monthly schedule.

16. **requirements.txt per Lambda group, not per Lambda.** 18 files for 39 Lambdas (grouped by shared deps) is the right granularity. Per-Lambda would be 39 nearly-identical files.

**The system is meaningfully more production-grade than it was 8 hours ago.** The grade movement from C+ to B+ on security and B- to B+ on reliability reflects real architectural improvement, not cosmetic changes. The original review's #1 recommendation — "stop building features, invest in foundations" — was followed, and the results are visible.
