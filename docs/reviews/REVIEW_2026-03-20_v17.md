## Executive Summary

Review #17 covers versions v3.7.47 through v3.7.82 — 35 releases that shipped a 12-page public website (averagejoematt.com), a public-facing AI Q&A endpoint (/ask/, /board/), a subscriber gate system, 2 new IC features (IC-29, IC-30), 5 completed Board Summit sprints, and significant operational hardening (in-memory rate limiting fix, nav standardisation). The platform has grown from an internal health intelligence tool to a public-facing product with live AI endpoints, which fundamentally changes the threat model and operational posture.

**Overall assessment: A-**

The grade drops from A to A- not because of regression, but because the platform has crossed a critical threshold — public exposure with AI endpoints — and the security, observability, and operational controls have not fully caught up to the new risk surface. The engineering quality remains excellent for a solo-developer project; the concern is that the system's *ambition* has outpaced its *defensive posture* in specific areas documented below.

**Defining story of this delta:** The platform successfully shipped a public website with live API endpoints, AI-powered Q&A, and subscriber management — all while maintaining cost discipline (~$13/month) and architectural coherence. This is a genuine product launch, not a dashboard. The Board Summit process produced focused, well-prioritized work. The sprint execution was disciplined.

**Key concerns (3):**
1. **Public AI endpoints without WAF or robust rate limiting** (R17-F01, Critical) — The site-api Lambda serves AI responses to the public internet with only in-memory rate limiting, which resets on cold start. No WAF, no persistent rate store, no abuse detection.
2. **Single-region architecture with cross-region split** (R17-F05, High) — Site-api in us-east-1, MCP + data in us-west-2, creates an operational split-brain risk and latency for the public API reading DynamoDB cross-region.
3. **In-memory rate limiting is not rate limiting** (R17-F02, High) — Lambda cold starts reset all rate counters. Under sustained load, every new Lambda instance starts with a clean slate.

## What the System Does Well

### 1. Architectural Coherence (Priya, Marcus)
The three-layer architecture (Ingest → Store → Serve) is clean, well-documented, and consistently applied. The Compute→Store→Read pattern (ADR-015) is a genuinely good architectural decision that separates concerns, enables independent retry, and supports multiple consumers. The single-table DynamoDB design (ADR-001) with explicit SOT domain ownership (ADR-004) is well-reasoned for the use case.

### 2. IAM Discipline (Yael, Marcus)
49 dedicated, least-privilege IAM roles — one per Lambda — all CDK-managed. This is exceptional for a solo-developer project. The `role_policies.py` centralisation pattern with `_ingestion_base()` helper is clean and auditable. No role has `dynamodb:Scan`. The site-api role is genuinely read-only (no PutItem, no Secrets access — confirmed by the v3.7.82 AccessDeniedException fix which proved the constraint works).

### 3. Statistical Rigour (Henning)
The INTELLIGENCE_LAYER.md "Known Statistical Limitations" section is production-grade documentation. BH FDR correction across 23 correlation pairs, n-gating that downgrades labels at low sample sizes, log-transform for HRV z-scores, the explicit "What a New Engineer Must Not Do" section — these show genuine statistical discipline. The correlative-not-causal framing (AI-2) throughout all prompts is correctly maintained.

### 4. ADR Discipline (All)
33 ADRs covering every significant decision. The ADRs are well-structured, include alternatives considered, and most importantly, document *non-decisions* (ADR-030 Google Calendar retirement, ADR-025 composite_scores removal). The ADR-032/033 S3 protection after the bucket wipe incident shows correct incident response.

### 5. Cost Engineering (Dana)
~$13/month for 49 Lambdas, 19 data sources, 16 IC features, 4 CloudFront distributions, a 12-page website, and AI-powered Q&A. The viral defence model (CloudFront TTL tiers → ~12 Lambda calls per endpoint at 50k visitors) is a genuinely clever cost architecture. The $0.33 cost estimate at 50k hits is credible.

### 6. CI/CD Pipeline (Jin)
The ci-cd.yml is well-designed: OIDC federation (no long-lived keys), lint → plan → deploy (manual approval) → smoke test → auto-rollback on failure → SNS notification. The lambda_map.json change detection is smart. The rollback-on-smoke-failure job (TB7-25) is a strong safety net.

### 7. Test Suite (Elena)
853 passing tests, 8 static linters, 10 integration tests (I1-I10) against live AWS. The MCP registry linter (test_mcp_registry.py) with R1-R5 checks was born from a real crash and directly prevents recurrence. The test_integration_aws.py design principles — every test catches a specific class of historical incident, read-only, <60s — are correct.

## Key Architectural Concerns

### Concern 1: Public AI Endpoints Are Under-Protected
The /api/ask and /api/board_ask endpoints serve Claude Haiku responses to the public internet. The rate limiting is in-memory (`_ask_rate_store`, `_board_rate_store`) — module-level dicts that reset on every Lambda cold start. Under sustained load, Lambda scales out and each new instance starts with empty rate stores.

### Concern 2: Cross-Region Architectural Split
Site-api runs in us-east-1 (for CloudFront + ACM), MCP + DynamoDB + all data lives in us-west-2. Every site-api call reads DynamoDB cross-region. This adds ~60ms latency per request and creates an operational split — CloudWatch logs, alarms, and debugging are split across two regions.

### Concern 3: 95 MCP Tools Approaching Cognitive Limit
SIMP-1 Phase 2 (target ≤80) is planned for ~April 13 but hasn't started. At 95 tools, the tool discovery surface in Claude's context window is large. This affects Claude's ability to select the right tool and increases cold-start time for the MCP Lambda.

## Detailed Findings

### R17-F01: Public AI Endpoints Lack Persistent Rate Limiting
**Severity:** Critical | **Category:** Security / Cost Protection
**Observed:** `site_api_lambda.py` uses in-memory dicts for rate limiting. These reset on cold start.
**Why it matters:** Lambda auto-scales under load. Each new instance gets empty rate stores. An attacker can exhaust the Anthropic API budget.
**Recommended change:** WAF rate-based rules on CloudFront distribution (board decision: approved).
**Effort:** M | **Confidence:** High | **Evidence:** Direct

### R17-F02: In-Memory Rate Limiting Resets on Cold Start
**Severity:** High | **Category:** Reliability / Security
**Observed:** v3.7.82 correctly moved to in-memory dicts (read-only IAM constraint) but created cold-start reset vulnerability.
**Recommended change:** WAF as primary layer, in-memory as secondary. Board decision: approved.
**Effort:** S | **Confidence:** High | **Evidence:** Direct

### R17-F03: No WAF on Public-Facing CloudFront Distributions
**Severity:** High | **Category:** Security
**Observed:** No WAF WebACL on any of the 4 CloudFront distributions.
**Recommended change:** WAF WebACL + rate-based rules. Board decision: approved (+$7/mo, Matthew approved).
**Effort:** S | **Confidence:** High | **Evidence:** Direct

### R17-F04: Subscriber Email Verification Has No Rate Limit
**Severity:** Medium | **Category:** Security
**Observed:** POST /api/subscribe and GET /api/verify_subscriber have no rate limiting.

### R17-F05: Cross-Region DynamoDB Reads Add Latency and Operational Complexity

## Final Verdict

**Grade: A-**

The Life Platform is a genuinely impressive personal engineering achievement. The grade drops from A to A- because the platform now has public-facing AI endpoints and the defensive controls haven't fully caught up. The critical path: harden public endpoints (WAF + rate limiting) and add the privacy policy before DIST-1.

**CTO challenge (Raj):** "Stop building features and start acquiring users. Write the story, harden the endpoints, and ship DIST-1."
