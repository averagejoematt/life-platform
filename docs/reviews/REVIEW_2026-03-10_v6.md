## Executive Summary

The platform has crossed an inflection point. Six of ten dimensions are now at A or A-, and the remaining four are within striking distance. The v3.4.4–v3.4.9 sprint was predominantly **correctness work** — fixing S3 path permissions, wiring missing KMS grants, consolidating secrets, cleaning CloudWatch alarm sprawl, and resolving a TLS certificate gap. This is the kind of unglamorous work that moves a platform from "works for the builder" to "works reliably." The incident log now has 25 entries — that's not a sign of fragility, it's a sign of honest operational culture.

**Overall grade: A-** (up from A- in Review #5, but with meaningful progress within the band)

## Board Member Assessments

### Priya Nakamura — Principal Cloud Architect
*"Is the system shape right?"*

**Grade: A.** The three-layer architecture (Ingest → Store → Serve) is clean, well-documented, and hasn't needed structural changes since Review #3. The CDK 8-stack decomposition maps logically to functional boundaries. Cross-stack references (table, bucket, DLQ, alerts_topic) are passed explicitly — no hidden coupling.

The compute→store→read pattern (ADR-015) continues to prove its value: daily-insight-compute, daily-metrics-compute, character-sheet-compute all pre-compute before the Daily Brief reads their outputs. This decouples intelligence computation from email delivery.

**One concern:** The `ARCHITECTURE.md` header says "41 Lambdas" but the Infrastructure Reference lists 39 under the Lambdas section (13+8+5+14=40, plus 2 Lambda@Edge = 42). The inventory in the bundle's source code section shows 57 `.py` files in `lambdas/`. This count drift is cosmetic but symptomatic of documentation that's hard to keep accurate as the platform grows. Consider auto-generating the Lambda count from CDK stack definitions.

### Marcus Webb — AWS Serverless Architect
*"Is this the right AWS implementation?"*

**Grade: A-.** The v3.4.6–v3.4.7 sprint exposed a class of bugs called **"CDK shadow state"** — env vars that existed only in the AWS console (EMAIL_RECIPIENT, EMAIL_SENDER) and were wiped when CDK took ownership. The fix was correct (added to `lambda_helpers.py` base env vars + `cdk.json` context), but the pattern should be a formal checklist item: *before any CDK stack deploys for the first time, audit all Lambda env vars via `aws lambda get-function-configuration` and ensure CDK definitions are complete.*

The KMS fix in v3.4.6 was significant — 13 functions were missing `kms:Decrypt`/`kms:GenerateDataKey`. This is a consequence of adding CMK encryption to the DynamoDB table without a systematic sweep of all readers. The `_ingestion_base()` helper now includes KMS unconditionally, which is the right fix. The `_email_base()` helper was also patched. **Verify that `_compute_base()` and `_operational_base()` (if they exist) also include KMS unconditionally.**

The EventBridge Scheduler migration (to `America/Los_Angeles` timezone) was a smart move — eliminates the DST cron-adjustment ceremony. But the bundle still references `deploy/deploy_dst_spring_2026.sh` in the Architecture doc. Clean up the stale DST reference.

**S3 path corrections (v3.4.7):** `raw/{source}/*` → `raw/matthew/{source}/*` across 6 ingestion functions. This is a serious finding — it means gap-fill attempts were silently failing due to AccessDenied on S3 writes. The fix is correct. However, this raises the question: were there S3 write failures in CloudWatch logs that went unnoticed? If so, the alarm coverage for S3 AccessDenied errors may need tightening.

### Yael Cohen — Cloud Security + IAM Architect
*"How could this fail or be exploited?"*

**Grade: A.** Promoting from A- based on three concrete improvements:

1. **S3 path permissions now user-prefixed** across all ingestion roles — no Lambda can write outside `raw/matthew/`. This was a real gap that could have caused data pollution in a multi-user scenario.
2. **Strava OAuth PutSecretValue** was missing — the token refresh cycle would have failed on the next OAuth expiry, causing a silent ingestion outage. Caught during the IAM health check, not by an incident. That's the kind of proactive finding that earns an upgrade.
3. **Secret consolidation (COST-B)** merges 4 ingestion keys into `life-platform/ingestion-keys`. The old secrets are on scheduled deletion (~2026-04-10). The migration path is clean: `setup_ingestion_keys.sh` handles the merge, CDK env vars already deployed.

**Remaining concerns:**
- Three pre-SEC-1 Lambdas still share roles (`anomaly-detector` → `life-platform-email-role`, `character-sheet-compute` → `life-platform-compute-role`, `dashboard-refresh` → `lambda-mcp-server-role`). These are documented as deferred, but they're the last vestiges of shared IAM. Each shared role grants broader permissions than the Lambda actually needs.
- `life-platform/api-keys` pending permanent deletion ~2026-04-07. Verify no Lambda has a cached env var pointing to it — the todoist incident (v3.4.8) showed this exact pattern.
- Webhook auth (`health-auto-export-api`) still uses Bearer token from the legacy `api-keys` bundle. After deletion, verify the webhook Lambda reads from the correct replacement secret.

### Jin Park — SRE / Production Operations Lead
*"What breaks at 2 AM?"*

**Grade: A-.** The alarm consolidation (87→~41) is excellent — alert fatigue is a real killer, and cutting alarm count in half makes each alarm more meaningful. The orphan deletion approach (audit first, then delete) is the right pattern.

The incident log now has 25 entries and clearly shows the incident pattern categories (deployment errors > stale config > wrong component). The v3.4.4 retroactive incident additions (5 entries from Review #4 findings) show intellectual honesty — backdating incidents isn't inflating the count, it's completing the record.

**CloudFront TLS (v3.4.9):** This was a user-visible outage — all three web URLs returning `ERR_CERT_COMMON_NAME_INVALID`. Root cause: `CERT_ARN_* = None` in `web_stack.py`. The fix (wiring actual cert ARNs with `ViewerCertificateProperty`) is correct and includes `TLSv1.2_2021` minimum protocol. **But:** how was this deployed without testing? A smoke test that verifies HTTPS connectivity to each distribution would have caught this before users noticed. Consider adding to the post-CDK-deploy smoke checklist.

**EMAIL_RECIPIENT wipeout (v3.4.6):** This is a sneaky class of bug — console-only config silently overridden by CDK. The "today's 9 AM brief was missed" impact was real. The fix (adding to `cdk.json` context) is architecturally sound. **Learning to codify:** Any CDK deploy that touches Lambda env vars should be followed by a `get-function-configuration` diff check.

### Omar Khalil — Data Architect
*"Is the data model coherent?"*

**Grade: A.** The single-table DynamoDB design is proven at this point — 27 partitions, no GSI, all access patterns served by PK+SK queries. The schema versioning (DATA-1) and ingestion validation (DATA-2) are both fully deployed.

The `item_size_guard.py` (300KB warn / 380KB truncate) is a smart defensive measure given the 400KB DynamoDB limit. Strava activities and MacroFactor food_log are the highest-risk partitions.

### Anika Patel — AI/LLM Systems Architect
*"Is the intelligence layer trustworthy?"*

**Grade: B.** Unchanged — and this is the dimension holding the platform back from all-A.

The core issue remains: **"built but not wired."**

- `ai_output_validator.py` exists (497 lines, well-designed BLOCK/WARN/PASS tiers) but is only wired into `daily-brief`. Six other email Lambdas (`weekly-digest`, `monthly-digest`, `wednesday-chronicle`, `nutrition-review`, `weekly-plate`, `monday-compass`) generate AI coaching output without output validation.
- `platform_logger.py` (structured logging) is only wired into `daily-brief`. The other 40 Lambdas still use stdlib `logging.getLogger()`, meaning CloudWatch Logs Insights queries can't correlate across Lambdas.

**Why this matters:** The AI output validator is the safety net between the LLM and the user's inbox. Without it, a hallucinated exercise recommendation during low recovery could slip through on a Wednesday Chronicle or Weekly Digest. The Daily Brief has coverage; the other six email Lambdas don't. That's an asymmetric safety posture.

**Positive note:** IC features are solid — 15 live IC features, hypothesis engine with validation rules, progressive context injection, attention-weighted prompt budgeting. The intelligence architecture is sophisticated. It's the operational safety wrapper that's incomplete.

**To reach A-:** Wire `ai_output_validator.py` into all 7 email Lambdas (not just daily-brief). This is the single highest-leverage remaining task.

### Henning Brandt — Statistician / Quantitative Methods
*"Are the conclusions actually valid?"*

**Grade: B.** The correlative language migration (AI-2) is complete and enforced in `ai_calls.py`. The Red Team / Contrarian pass (IC-17) adds a structural check against confirmation bias. Data quality scoring (IC-24) flags incomplete sources before AI calls.

**Ongoing concern:** The platform generates habit×outcome correlations from 7-day windows (daily-insight-compute). With N=1 data and 7 observations, any correlation is fragile. The hypothesis engine requires 7 days minimum sample and 3 confirming checks for promotion — these are reasonable guardrails, but the user should see confidence intervals or sample-size caveats alongside correlation claims. Currently, the prompt instructs correlative framing ("associated with," not "causes"), but doesn't expose the statistical fragility to the reader.

**No regression this cycle.** The AI-2 language migration holds.

### Raj Srinivasan — Technical Founder / Startup CTO
*"What's the wedge and where are you fooling yourself?"*

**Grade: B+.** The platform is an impressive personal infrastructure achievement. The architecture review process itself (6 reviews, formal grades, Jira-style tracking) is more rigorous than most startups manage.

**Where you might be fooling yourself:**
1. **The "built but not wired" gap has been identified for 3+ reviews.** If it's important enough to flag every review, it's important enough to spend a Saturday wiring it. If it's not important enough to spend a Saturday on, stop flagging it and accept the risk explicitly.
2. **Documentation richness vs. utility:** 21 root docs, 14 archived docs, 8 review files. The documentation is thorough but may be outpacing the pace at which it's consumed. Who reads `DATA_DICTIONARY.md` other than the review bundle generator? Consider which docs earn their maintenance cost.
3. **Feature velocity has been near-zero since v3.3.x.** The hardening sprint was necessary, but the platform's value comes from what it tells Matthew about his health — not from how clean the IAM policies are. After this review, consider a feature sprint (Google Calendar integration, Brittany email, grip strength tracking).

## Board Brainstorm — New Ideas

1. **`deploy/post_cdk_smoke.sh`** — After any `cdk deploy`, automatically hit each CloudFront URL + invoke a test MCP call + verify Daily Brief Lambda isn't in error state. Would have caught the TLS bug (v3.4.9) and the EMAIL_RECIPIENT wipeout (v3.4.6).

2. **CDK env var diff check** — Before deploying a CDK stack that manages Lambdas, automatically run `aws lambda get-function-configuration` for each Lambda in the stack and diff against CDK-defined env vars. Flags console-only env vars that CDK would overwrite.

3. **Automated Lambda count in ARCHITECTURE.md** — Add a small script (or CDK Aspect) that emits the actual Lambda count from stack definitions into the doc header. Eliminates manual count drift.

4. **AI output validator as Lambda Layer middleware** — Instead of wiring `validate_ai_output()` into each Lambda individually, make it a decorator or middleware in the shared Layer that wraps all `call_anthropic()` invocations. One integration point, all Lambdas covered.
