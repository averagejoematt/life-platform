**Review Date:** 2026-03-08

## PHASE 0: SCOPE & ASSUMPTIONS

**What this system is:** A single-user personal health intelligence platform on AWS. 19 data sources ingested into a single DynamoDB table, surfaced through 144 MCP tools to Claude, with 9 automated email digests, 3 web properties, a gamified character sheet, an "intelligence compounding" (IC) layer, and an AI "Board of Directors" coaching system.

**What it aspires to be:** Eventually a commercial-grade platform (open-source framework, multi-user, productization).

**Key assumptions made for this review:**
- No IaC (Terraform/CDK) — all infrastructure managed via bash deploy scripts
- No CI/CD pipeline — manual local deploys
- No automated tests beyond a single qa-smoke Lambda
- Single user, single region, single account
- The builder is both developer and sole operator

**Missing artifacts that would improve confidence:**
- IAM policy JSON (actual role definitions, not descriptions)
- Lambda environment variable inventory (actual, not documented)
- CloudWatch alarm configurations
- Actual board_of_directors.json content
- Full ai_calls.py (prompt templates)
- Network/VPC configuration (or confirmation of no VPC)
- API Gateway configuration details
- S3 bucket policies and encryption settings

## WHAT THE SYSTEM DOES WELL (Design Decisions to Preserve)

1. **Single-table DynamoDB design.** The `USER#matthew#SOURCE#<source>` / `DATE#YYYY-MM-DD` key pattern is clean, scalable, and avoids GSI cost. All access patterns are PK+SK queries. This is textbook.

2. **Compute → store → read pattern.** Pre-computing character sheet, daily metrics, insights, and adaptive mode before the Daily Brief is architecturally sound. Decouples compute from presentation. This is the right pattern.

3. **Cost consciousness.** Every feature has a cost estimate. The $10/month for this capability set is remarkable. Lambda Function URLs instead of API Gateway for MCP, on-demand DDB, Haiku for most AI calls — all correct tradeoffs.

4. **Source-of-truth domain ownership.** The explicit SOT mapping in the profile record (Whoop for sleep, Strava for cardio, etc.) is exactly how data conflicts should be resolved. Config-driven, not code-driven.

5. **Gap-aware self-healing backfill.** Checking the last 7 days of DDB on each run and filling gaps automatically is elegant. No external state tracker needed.

6. **Board of Directors externalization.** Moving expert personas to S3 config with a shared `board_loader.py` and fallback is clean separation of content from code.

7. **Documentation quality.** ARCHITECTURE.md, SCHEMA.md, RUNBOOK.md are genuinely useful. The incident log captures patterns. This is better documentation than most production systems I've reviewed.

8. **Three-tier webhook source filtering.** The explicit tier classification (Apple-exclusive / cross-device filtered / skip) for Health Auto Export prevents cross-source contamination.

9. **EventBridge Scheduler migration.** Moving to timezone-aware scheduling eliminates DST manual intervention. Good operational hygiene.

10. **PITR + deletion protection on DynamoDB.** Non-negotiable for a data platform and it's done.

## KEY ARCHITECTURAL CONCERNS

### Concern 1: The "Shared Role" Anti-Pattern (Critical)

`lambda-weekly-digest-role` is shared by **10+ Lambdas** including the Daily Brief, all digests, all compute Lambdas, the hypothesis engine, and QA smoke tests. This role has:
- DynamoDB read/write
- Secrets Manager access to `life-platform/api-keys` (contains Anthropic, Todoist, Habitify, Notion, Dropbox keys)
- SES SendEmail
- S3 PutObject on dashboard and buddy paths

**Why it matters:** Any vulnerability in any of these 10 Lambdas gives an attacker full access to all secrets, all data, and email sending capability. The blast radius is the entire platform. A bug in the QA smoke Lambda could exfiltrate all API keys.

**Impact if ignored:** A single compromised dependency in any of these Lambdas can access every upstream API credential in the consolidated secret.

### Concern 2: No Infrastructure as Code (High)

~170 bash scripts in `deploy/` with no IaC (Terraform, CDK, SAM, CloudFormation). Infrastructure state exists only in AWS and in the operator's memory.

**Why it matters:** You cannot recreate this environment from scratch. You cannot audit drift. You cannot version infrastructure changes. A future team cannot reason about infrastructure without you.

### Concern 3: No CI/CD or Automated Testing (High)

Deploys are manual. The sole test is a `qa-smoke` Lambda that is invoked on-demand. No unit tests, no integration tests, no pre-deploy validation.

**Why it matters:** The incident log shows 5 deployment-related outages in 2 weeks (wrong handler, wrong binary, missing IAM, wrong zip filename). This rate proves manual deployment is the system's primary failure mode.

### Concern 4: 144 Tools / 35 Lambdas Complexity Budget (High)

For a single-user platform processing ~19 data sources, 144 MCP tools and 35 Lambdas is an extremely high surface area. Each tool and Lambda is a maintenance liability, a potential failure point, and a documentation burden.

**The honest question:** How many of these tools are used more than once a month? The answer determines whether this is a platform or a feature graveyard.

### Concern 5: AI Health Coaching Without Clinical Guardrails (High)

The system generates health advice ("training recommendations", "nutrition review", "protocol suggestions") using LLM prompts that reference medical data (blood panels, glucose, blood pressure, genome SNPs). There are no clinical guardrails, disclaimers, or validation loops.

**Why it matters:** An LLM hallucinating about medication interactions, supplement dosing, or exercise intensity for someone at 302 lbs with health risk factors is a safety concern, not just a quality concern.

## THE SKEPTICAL PRINCIPAL ENGINEER PASS

### 15 Harshest But Fairest Critiques

**1. This is a solo hobby project with enterprise-grade documentation pretending to be a platform.** The sophistication is in the docs, not in the operational maturity.

**2. 144 tools is a vanity metric.** Each tool adds maintenance surface area. One broken tool is invisible until someone calls it.

**3. The shared IAM role violates the platform's own stated principles.** ARCHITECTURE.md says "dedicated, least-privilege" then admits 10 Lambdas share one role.

**4. The "Intelligence Compounding" layer is building a cathedral on quicksand.** IC features assume data is correct. The incident log proves it isn't always.

**5. The deploy/ directory is the real architecture and it's chaos.** 170+ scripts, many one-time fixes.

**6. The incident rate is a leading indicator, not a lagging one.** 14 incidents in 14 days is structural, not incidental.

**7. The AI health coaching has no safety rails.** Exercise recs and supplement timing for a 302lb individual with genome-flagged risks. No disclaimers.

**8. "Config-driven" is only as good as config validation.** No schema validation on S3 config files.

**9. The "compute → store → read" chain has no error propagation.** Failed compute = stale data consumed silently.

**10. Multi-tenancy is not an evolution — it's a rewrite.** `USER_ID = "matthew"` in every Lambda.

**11. The AI cost is underestimated.** IC features will push AI from $3 to $8-15/month.

**12. The module coupling is hidden by copy-paste deployment.** Version mismatch between Lambda zips is invisible and inevitable.

**13. The Anthropic API is a single point of failure with no fallback.** Regional outage at 10 AM = no Daily Brief, no enrichment, nothing.

**14. S3 static site writes have no concurrency protection.** Daily Brief and dashboard refresh could race.

**15. The "North Star" is philosophical, not measurable.** No metric for "reducing the gap between knowing and doing."

## FINAL VERDICT

This is a genuinely impressive personal engineering project. The architectural thinking is strong. The documentation is excellent. The cost consciousness is admirable. The feature velocity is remarkable.

But the system is suffering from **feature-obsession at the expense of operational maturity.** You've built v2.91 with 144 tools in roughly 2 weeks. That velocity came at a cost: no tests, no CI/CD, no IaC, shared IAM roles, copy-paste deployments, and 14 incidents.

**The most important thing you can do next is NOT build another feature.** It's to invest 2-3 sessions in operational foundations: IAM decomposition, Lambda Layers, basic CI/CD, and input validation. These unglamorous changes will reduce your incident rate, make the system operable by someone other than you, and create a foundation that can actually support the IC intelligence features without compounding technical debt.

Fix the plumbing before you install the smart home system.
