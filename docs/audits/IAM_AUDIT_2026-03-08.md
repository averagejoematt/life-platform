## FALSE POSITIVES (no action needed)

### cloudwatch:PutMetricData with * resource
**Flagged on:** macrofactor, strava, canary, anomaly-detector
**Why it's fine:** CloudWatch PutMetricData genuinely requires `*` as the resource — AWS doesn't
support resource-level permissions for this action. This is correct and expected.
**Action:** None. Will suppress in future audit script.

### dynamodb:Scan on lambda-data-export-role
**Why it's fine:** Data export Lambda does a full table dump monthly — Scan is intentional.
**Action:** None.

### dynamodb:DeleteItem on lambda-canary-role
**Why it's fine:** Canary writes a test record and then deletes it (cleanup). Intentional.
**Action:** None.

### Broad log group ARN pattern (weather, notion, dropbox, habitify, freshness-checker)
**Pattern:** `arn:aws:logs:us-west-2:205930651321:log-group:/aws/lambda/<name>:*`
**Why it's fine:** This is the correct scoped pattern for Lambda logging permissions — it's
Lambda-specific, not a wildcard. The audit script over-flagged trailing `:*` on log groups.
**Action:** None. Will fix audit grep to exclude log group ARNs.

## REAL ISSUES — LOW PRIORITY (fix in next session)

### ses:SendEmail with * resource (dlq-consumer, anomaly-detector, canary)
**Risk:** Could send email from any SES identity, not just mattsusername.com
**Fix:** Scope to `arn:aws:ses:us-west-2:205930651321:identity/mattsusername.com`
**Scripts to update:** deploy_rel2_dlq_consumer.sh, deploy_anomaly_detector.sh, deploy_rel4_canary.sh
**Priority:** Low (single-account, single identity — minimal blast radius)

## REAL ISSUES — DEFER (already addressed by SEC-2)

**Flagged on:** health-auto-export, notion-journal-ingestion, dropbox-poll,
               habitify-data-ingestion, todoist-data-ingestion, mcp, monday-compass, canary
**Status:** SEC-2 updated notion/dropbox/todoist Lambda env vars to point to domain secrets.
The IAM *policies* on these roles still list api-keys in their allowed resource ARNs.
**Full fix:** Update each role's `api-keys-read` inline policy to point to the specific
domain secret ARN instead. Lower urgency now that env vars are updated — the Lambdas
won't read from api-keys even if the policy allows it.
**Priority:** Defer to next hardening session (IAM-1 follow-up).
