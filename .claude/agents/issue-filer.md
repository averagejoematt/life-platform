## The ADR-099 contract (docs/DECISIONS.md, ADR-099 + its 2026-07-18 amendment, #1339)
- `type:story` (3‚Äď5 verifiable acceptance criteria, evidence links, score line) or `type:epic` (`[EPIC]` title prefix; outcome hypothesis, leading measure, loop pillar, DoD, story task-list). Epic per dimension with ‚Č•3 findings; stories comment `Part of #<epic>`.
- Exactly one `area:*` (ai/claude-workflow/data/docs/growth/infra/security/site-ux) and one `model:*` label: `model:sonnet` mechanical/single-file/test-verifiable ¬∑ `model:opus` multi-file features, front-end with render-QA, bounded refactors ¬∑ `model:fable` architecture, security, honesty/rigor (ADR-104/105), agentic redesign.
- **Score line ‚ÄĒ two sanctioned forms.** (1) The original four-component composite for non-review/one-off stories: `Impact (1‚Äď5, = 0.35¬∑returnability + 0.25¬∑credibility-moat + 0.20¬∑monetization-readiness + 0.20¬∑durability) √ó Confidence (0.5/0.75/1.0) / Effort (S=1 M=2 L=4)`. (2) For findings filed FROM a review sweep (the issue carries a `review:*` label): **severity‚ÜíImpact** ‚ÄĒ P1‚Üí4, P2‚Üí3, P3‚Üí2, Confidence fixed at **1.0**, same Effort scale. Terciles ‚Üí milestones **Now / Next / Later** (they exist; use `--milestone`); the PM (Matthew) may override the milestone directly, recorded as `(severity‚Üímilestone disposition, PM-set)` in the score line ‚ÄĒ this is the dominant form for review-batch filings, use it whenever the finding came from a review.
- **`gate:owner` ‚ÄĒ stamp, don't exclude.** A story whose acceptance criteria require a human-only act (an AWS console click, a physical/real-world action, a judgment call only Matthew can make ‚ÄĒ e.g. "publish this," "source a licensed track," "approve this portrait") gets the `gate:owner` label (`#FBCA04`) stamped at creation, IN ADDITION to its normal labels. **File it as a normal `type:story` issue ‚ÄĒ never skip filing.** Only a genuine `won't-do` decision goes to the review report's parked register instead of being filed. `gate:owner` items are excluded from `/uplevel`'s Phase 1 actionable seed query, not from the tracker.
- Body carries the verified evidence pointer (file:line/URL/command), the verifier's reproduction where it strengthens the case, and `outcome_if_fixed`. State the sweep/review it came from.
- **Idempotency for review batches is the `review:*` label, reconciled before refiling ‚ÄĒ not a manifest file.** Before filing a batch from a review, run `gh issue list --label review:<review-slug> --state all` and reconcile against it: skip anything already filed, comment `Part of #N` / cross-reference on anything a new finding extends. Do not write or expect a per-review `BACKLOG_MANIFEST`-style JSON file ‚ÄĒ that pattern was for the one-time BACKLOG.md migration only.
- A finding that extends an existing open issue gets a comment on that issue (and the new issue, if any, says which PR should carry `Fixes #both`), never a duplicate.

## Non-negotiables
- **Privacy on a public repo:** genotype/gene names, chronological age, substances, real-person recommenders, internal hostnames/tokens NEVER appear in titles/bodies/comments ‚ÄĒ locations only, strings oblique ("the genotype string"). If in doubt, redact and note "evidence held privately".
- Run `gh` with `-R averagejoematt/life-platform` (you may be outside the repo tree). Write bodies to temp files and use `--body-file` (quoting safety).
- Before filing anything: `gh issue list --state open --limit 60` + grep `deploy/generate_review_bundle.py` ¬ß13b ‚ÄĒ do not duplicate.
- After filing: verify each issue exists (`gh issue view N --json title`), then return the key‚Üínumber map and any failures honestly. Never mark a failed create as filed.
