## NEXT SESSION INTENT — Architectural Review

**Do NOT start building new features.** The explicit goal for next session is a holistic pause to assess the system across 5 dimensions before growing scope further:

### 1. Security
- IAM role permissions — are they least-privilege? Any over-scoped roles?
- API key management — is the MCP API key rotation strategy sufficient?
- Secrets Manager usage — are all secrets properly scoped?
- Lambda execution roles — any cross-role risks?
- S3 bucket policy — public access blocked? encryption at rest?
- DynamoDB — deletion protection on? any access control gaps?

### 2. Performance
- Lambda cold start latency — is the current warm-up strategy (nightly EventBridge) sufficient?
- `parallel_query_sources` — ThreadPoolExecutor with max_workers=5, is this optimal?
- DynamoDB read patterns — are we paginating efficiently? Any hot partitions?
- Memory cache (10min TTL) vs DynamoDB cache (26hr TTL) — is this layering correct?
- Largest data sources (Apple Health ~3000+ items) — any query timeout risks?
- MCP bridge response time — are there p95/p99 latency concerns?

### 3. Optimization
- Cost audit — are we still under $25/month? What are the biggest cost drivers?
- Lambda memory/timeout settings — are they tuned per-function?
- DynamoDB capacity mode — on-demand vs provisioned, any savings possible?
- S3 storage classes — raw data lifecycle policies?
- Cache hit rates — are the nightly warmer pre-computations actually being used?
- `lean=True` stripping — is this consistently applied where it should be?

### 4. Schema
- Single-table design review — are the PK/SK patterns still optimal as data grows?
- Field naming consistency — any inconsistencies across sources?
- The `source_of_truth` addition to profile — does this suggest a broader profile v2?
- Missing fields — are any sources under-capturing available data?
- `CACHE#matthew` partition — is this getting large? any TTL cleanup needed?
- Apple Health data shape — is the XML → DynamoDB mapping losing useful fields?

### 5. Architectural Design
- Source-of-truth model — now implemented, are there edge cases not covered?
- The warmer Lambda — should it be a separate function vs bundled in mcp_server?
- Error handling — are Lambda failures surfaced anywhere? alerting?
- Observability — any CloudWatch alarms or dashboards? what's the on-call story?
- Multi-user future — is the `USER#matthew` prefix sufficient scaffolding?
- Backup strategy — DynamoDB point-in-time recovery enabled?
- The `get_sot()` function calls `get_profile()` on every invocation — profile cache is warm-instance only, is this a cold start concern?
