### Presenting problem
Matthew woke to a flooded inbox of AWS alarm emails: repeated `canary-ddb-failure`, `canary-s3-failure` (ALARM + OK pairs every 4 hours), plus a `dlq-depth-warning` at 6:10 AM.

### Fix 1 ŌĆö v3.6.5: Canary IAM policy (root cause: day-1 misconfiguration)

**Problem:** `operational_canary()` in `role_policies.py` was written as a read-only policy, but the canary Lambda is a write-read-delete round-trip health check. Every single canary run since deployment (~2026-03-10) was failing with `AccessDeniedException`. The alarm fires on failure and auto-clears on missing data, producing ALARM+OK pairs every 4 hours.

Additional issues found:
- Secret reference was `life-platform/mcp-api-key` (nonexistent secret)
- Missing `kms:Decrypt` / `kms:GenerateDataKey` for the CMK-encrypted DDB table
- Missing `ses:SendEmail` (canary couldn't even send its own alert emails)
- `MCP_SECRET` default in `canary_lambda.py` pointed to `life-platform/api-keys` (being permanently deleted 2026-03-17)

**Fix:**
- Rewrote `operational_canary()` policy with correct write permissions
- Fixed `MCP_SECRET` default in `canary_lambda.py` ŌåÆ `life-platform/ai-keys`
- Applied policy directly to `CanaryRole` via `aws iam put-role-policy` (no CDK needed)
- Canary confirmed `ALL PASS Ō£ģ` (DDB 51ms, S3 171ms, MCP skipped/null)
- `deploy/canary_policy.json` committed for future reference

### Fix 2 ŌĆö v3.6.6: Missing layer on 13 ingestion Lambdas (root cause: CDK omission)

**Problem:** DLQ had 10 messages, all Todoist, all with error `'Logger' object has no attribute 'set_date'`. Investigation revealed `todoist-data-ingestion` had `Layers: null`. Full audit showed **all 13 ingestion Lambdas** (except Garmin, which was correctly wired) were missing the `life-platform-shared-utils:4` layer. Without the layer, `from platform_logger import get_logger` silently falls through to stdlib `logging.Logger`, which has no `set_date()`. The CDK `ingestion_stack.py` defined `shared_utils_layer` but only passed `shared_layer=shared_utils_layer` to Garmin.

**Fix:**
- Added `shared_layer=shared_utils_layer` to all 13 affected Lambdas in `ingestion_stack.py`
- Immediately attached layer to all 13 via `aws lambda update-function-configuration` (live, no redeploy)
- DLQ purged (10 stale messages)
- Note: `platform_logger` failures were silently swallowed by `except ImportError` ŌĆö this is why it wasn't caught earlier. The fallback logger works fine until something calls `set_date()`.

**Lambdas fixed:** whoop, garmin (already had it), notion, withings, habitify, strava, journal-enrichment, todoist, eightsleep, activity-enrichment, macrofactor, weather, dropbox-poll, apple-health
