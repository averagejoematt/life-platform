## Key architecture decisions
- **Per-entry caffeine detection**: Unlike `get_nutrition_biometrics_correlation` which uses daily totals only, this tool scans individual `food_log` entries for `caffeine_mg > 0` and extracts the latest timestamp — this is the key insight that enables timing analysis
- **Same-date join**: MacroFactor date X caffeine → Eight Sleep date X sleep (same night)
- **Bucket-first, correlation-fallback**: The primary recommendation engine uses bucket comparison (more interpretable for the user). Pearson r is the fallback if bucket analysis doesn't show a clear threshold
- **Degradation threshold**: 2 percentage points drop in sleep efficiency triggers a cutoff recommendation. This is conservative but avoids false signals with limited data
- **Reuses existing helpers**: `query_source()`, `pearson_r()`, same `t2d()`/`d2hm()` time conversion pattern as `get_meal_timing`
