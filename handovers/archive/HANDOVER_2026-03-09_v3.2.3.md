## What Happened This Session

### 1. Layer zip structure root cause fixed
All previous Layer versions (v1–v3) used `zip -j` which stripped the `python/` directory prefix. Lambda Layers require files at `python/*.py` inside the zip. Fixed to `zip -r` in `deploy/p3_build_shared_utils_layer.sh`. Layer v4 published and verified.

### 2. COST-2 deployed ✅
EMF metrics in `mcp/handler.py` — `ToolInvocations`, `ToolDuration`, `ToolErrors` per tool, namespace `LifePlatform/MCP`. Zero IAM changes. SIMP-1 archiving gated on 30 days of data (~2026-04-08).

### 3. SIMP-2 Phase 1 fully working ✅
`weather_handler.py` running clean on Layer v4: 2 records, 0 errors, no DATA-2 warnings.

### 4. SIMP-2 Phase 2 assessed and closed
- **Strava:** range-based API (one call returns all activities in window) — forcing per-day callbacks would turn 1 API call into 7. Architecturally worse.
- **Garmin:** `native_deps: true` (`garminconnect`/`garth`) — must bundle in own zip, can't use Layer. Framework would have to be bundled inline, creating two diverging copies.
- **Decision:** stop migration. Framework stays in Layer for future new Lambdas fitting the per-day poll pattern. All existing Lambdas stay as-is.
- **Lesson documented:** Future architectural recommendations must validate per-source API shapes and dependency constraints upfront before committing to migration scope.

### 5. ci/lambda_map.json updated
- `weather_lambda.py` → `weather_handler.py`
- `ingestion_framework.py`, `ingestion_validator.py`, `item_size_guard.py` added to `shared_layer.modules`
- Removed those 3 from `skip_deploy`
